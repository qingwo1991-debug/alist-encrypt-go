## High-Level Background
- This repo is the Go rewrite of the older Node project at:
  - `/root/AI/alist-encrypt`
- Upstream service is OpenList / OpenList-compatible Alist backend.
- Primary product goals discussed in this session:
  - Web playback works reliably
  - WebDAV playback works reliably
  - No ciphertext passthrough on decrypt failure
  - Better first-frame speed / seek responsiveness
  - Weak-network and slow-upstream resilience
  - Background warmup/preheat is observable and provably useful

## What Was Fully Implemented

### 1. Redirect playback unification
- `/redirect/:key` was moved onto the same decrypt playback orchestration path as `/d` and `/dav`.
- This includes carrying display path and compat key context so redirect playback participates in the same strategy learning.

### 2. Fail-closed decrypt behavior
- Removed the dangerous behavior where decrypt failures could fall back to encrypted bytes.
- Current intent is:
  - decrypt required => fail with error if decrypt path fails
  - do not silently serve ciphertext

### 3. CI test gate
- GitHub Actions workflow was changed so tests run before Docker build/push.

### 4. File-level warmup observability
- `ProbeScheduler` now records file-level warmup/probe events.
- Already implemented metrics/records include:
  - specific file path
  - source
  - status
  - resolved size
  - failure reason
  - raw_url fetched
  - range probed
  - meta persisted
  - recent records list

### 5. Warmup consumer hit stats
- A first-pass `consumer_hit` mechanism was added.
- This records when a previously warmed file is actually consumed by playback.
- Current scenarios wired:
  - `http_download`
  - `webdav_get`
  - `redirect_playback`

### 6. Admin UI updates
- `enc-webui/src/views/setting-alist/index.vue` was updated to show:
  - queue overview
  - file-level warmup counts
  - source/status/failure summaries
  - recent warmup records
  - recent consumer-hit records
  - hit rate
- `enc-webui` was built and copied into `web/public`.

### 7. Partial invalidation / cleanup
- Some invalidation work was added:
  - delete / rename / move paths now clear warm-related state partially
  - playback failure paths start invalidating old warm success state in some cases

## Main Remaining TODOs

### A. Finish warm state reliability model
The following were planned but not fully completed:
- explicit warm state classification:
  - `warm_ready`
  - `warm_stale`
  - `warm_invalid`
- these states should be exposed clearly in stats/UI
- stale detection should be based on upstream staleness threshold
- invalidation should not just delete warm state silently; it should be visible as invalidated

### B. Finish remaining probe record fields
These fields were planned but not fully added end-to-end:
- `size_source`
- `used_auth_mode`
- `queue_wait_ms`
- `priority`
- `invalidated`
- `consumer_hit_count`
- `last_consumer_hit_at`

Notes:
- Some of these were partly started conceptually but not fully wired into the actual structs / stats / UI.
- Re-read the current `internal/handler/probe_scheduler.go` before editing; the file changed multiple times during the session.

### C. Strengthen invalidation / stale cleanup
Need a more complete strategy for:
- raw_url fetch returns 401/403/404
- HEAD or Range returns not found / forbidden
- playback returns `upstream_4xx`
- upstream object deleted, moved, permission changed, signature expired

Should invalidate:
- successful warm state
- raw_url cache
- size cache
- display/encrypted path mapping where appropriate
- preheat record status / invalidation event

### D. Consumer-hit dedupe and quality
This area needs a second pass:
- dedupe repeated hits for the same file within a short window
- optionally keep only one hit per TTL window
- expose hit counts per warmed file
- do not overcount repeated playback refreshes

### E. Node compatibility gaps not yet addressed
Still needs decision / implementation:
- `/images/*` passthrough compatibility
- whether to keep / reintroduce `/api/fs/put-back`
- cover image thumb / hide cover file behavior from old Node version
- WebDAV `PROPFIND 404/401` edge behavior review
- directory GET vs file GET distinctions

### F. Big-file and high-latency strategy work
Not finished:
- layered handling for files `> 10G` and `> 25G`
- large-file-specific warmup budgeting
- better host-level timeout/budget strategy using observed RTT
- provider-level observability for timeout / 4xx / 5xx trends
