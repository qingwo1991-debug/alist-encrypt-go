# 当前项目视频播放链路梳理

## 1. 当前播放主链
以 `5344` 播放一个文件为例，核心步骤是：

1. 前端先拿文件信息  
   入口通常是 `/api/fs/get` 或 `/api/fs/list`，把“明文路径”映射到“密文路径”，同时尽量拿到：
   - `raw_url`
   - `size`
   - 已缓存的 `V2` 元数据

2. 真正播放时进入解密播放编排  
   统一入口是 `internal/handler/playback_orchestrator.go` 里的 `executeDecryptPlayback(...)`。

3. 先尝试直接用缓存的解密元数据  
   如果 `FileDAO` 里已经有：
   - `ContentVersion`
   - `HeaderLen`
   - `NonceField`
   - `PlainSize`
   - `CiphertextSize`  
   就直接把这些塞进请求上下文，后面不用再探测。

4. 没有完整元数据时，再主动探测内容头  
   只对 `redirect` 和 `webdav` 这类播放场景做。  
   它会依次试：
   - 当前 `TargetURL`
   - `openalist /dav<encryptedPath>`
   - `openalist /d<encryptedPath>`  
   然后调用 `InspectEncryptedContent(...)` 去识别是不是 `V2`，并拿到：
   - `version`
   - `headerLen`
   - `plainSize`
   - `ciphertextSize`
   - `nonce`

5. 决定这次怎么读上游  
   先选播放策略，入口在 `internal/proxy/stream.go` 的 `SelectOptimalStrategy(...)`。  
   现在已经简化成只返回一个当前首选策略：
   - `range`
   - `chunked`
   - `full`

6. 走真正的下载解密  
   `executeDecryptPlayback()` 里调用 `StreamProxy.ProxyDownloadDecryptWithStrategyForStorage(...)`。  
   这里才是真正按：
   - legacy
   - V2  
   去构造 cipher、改写 Range、输出明文流。

7. 成功后回写缓存和学习状态  
   成功时会：
   - 记录最近播放 hint：`RecordPlaybackHint(...)`
   - provider 级策略学习：`RecordSuccess(...)`
   - 记录 size/playback success
   - 对首帧 GET 异步加入 warmup 队列

## 2. 解密具体发生在哪
真正“解密”不是在 `/api/fs/get`，而是在播放流输出时。

分两层：

- `元数据层`
  - 判断是旧格式还是 `V2`
  - 算清楚 `plainSize/ciphertextSize/headerLen/nonce`
- `流解密层`
  - 根据 Range 改写上游请求
  - 按 legacy 或 `V2` cipher 解流
  - 返回客户端看到的明文 `206/200`

所以 `预热` 和 `学习` 都不会提前把整个视频解开，它们做的是：
- 提前拿齐解密所需元数据
- 提前确定更合适的读取方式
- 避免每次播放都重新试错

## 3. 当前项目里不止一套“学习”

### A. 播放策略学习
这是现在收敛后的主学习，按 `provider` 学：
- 根服务端在 `internal/handler/stream_strategy.go`
- 移动端在 `mobile/openlist-lib/openlistlib/encrypt/strategy_selector.go`

它学的是：
- 这个云盘更适合 `range` / `chunked` / `full` 哪一种

它不学：
- `raw_url` 怎么拿
- `V2` 元数据是什么
- 某个文件的 nonce 是多少

### B. 最近播放 hint
这是更短期的热提示，在 `internal/proxy/stream.go`：
- `RecordPlaybackHint(...)`
- TTL 只有 2 分钟

它是“刚刚这个 targetURL + storageKey 用某策略成功了，短时间内直接复用”。

### C. Range 兼容学习
也是在 `internal/proxy/stream.go`。
它学的是：
- 这个上游到底支不支持 Range
- 不支持时多久后重探

### D. 根服务端目录级“元数据获取策略学习”
这和播放不是一回事，在：
- `internal/handler/proxy_strategy.go`
- `internal/handler/strategy_cache.go`

它学的是：
- 某个目录下，拿 file size/file info 更适合走：
  - `file_info_cache`
  - `file_size_cache`
  - `head_request`
  - `range_request`
  - `propfind`

### E. 移动端“远程文件大小探测策略学习”
在 `mobile/openlist-lib/openlistlib/encrypt/proxy.go`。  
它按 `encPathPattern` 学：
- `HEAD`
- `Range`
- `WebDAV PROPFIND`
哪种更适合拿远程 size

所以严格说，当前系统不是“一套学习”，而是：
- 一套播放策略学习
- 一套最近播放 hint
- 一套 range 能力学习
- 一套根服务目录级 size/info 学习
- 一套移动端路径级 probe 学习

## 4. 预热到底做了什么
预热主逻辑在 `internal/handler/probe_scheduler.go`。

它不是预解密视频，而是预热这几样：

- `raw_url`
- `size`
- `range` 探测结果
- `V2` 元数据是否已识别
- 这些结果是否已写回缓存

最关键的是：首帧成功后会异步触发 warmup。  
也就是说它的目标是：

- 第一次播放成功后
- 把同文件后续播放要用的元数据先补齐
- 让下一次不再走冷路径

## 5. 预热和学习是怎么配合的

### 预热
解决的是：
- “这个文件的播放前置资料有没有准备好”

它关心的是文件级/目标级数据：
- `raw_url`
- `size`
- `V2 meta`
- `nonce`

### 学习
解决的是：
- “拿到这些资料后，应该用什么方式去播更稳”

它关心的是 provider 级策略：
- `range`
- `chunked`
- `full`

所以正常配合关系应该是：

1. 预热先把文件资料准备好
2. 学习决定这次读法
3. 播放成功后再更新学习和缓存

## 6. 现在它们到底是在加速还是减速

### 预热本身，设计上是加速
如果预热命中正确，它能明显减少：
- `fs/get`
- `raw_url fresh fetch`
- `/dav fallback`
- `InspectEncryptedContent`
这些冷路径成本。

所以预热本意是加速，不是减速。

### 学习本身，大方向也是加速
尤其是 provider 级播放策略学习，设计上能减少：
- 每次都从 `range` 开始试
- 遇到明确不兼容还要重复试错

所以学习本意也是加速。

### 但当前实现里，确实存在“局部减速/重复试错”
从代码结构看，问题主要不是“预热错了”或“学习错了”，而是：

1. 学习层太多，不是一个统一系统  
   有些学播放，有些学 size，有些学 range，有些学 probe。

2. 学到的东西不一定覆盖最贵的冷路径  
   比如播放策略学会了 `chunked`，但如果每次还要重新：
   - 取 `raw_url`
   - fallback `/dav`
   - inspect `V2`
   那首帧还是慢。

3. 缓存失效/覆盖会让预热收益丢掉  
   比如轻量目录信息把完整 `V2 meta/raw_url` 覆盖掉，然后又回到冷路径重试。

所以更准确的判断是：

- 预热和学习的目标都是加速
- 当前确实有反复试错
- 反复试错主要不是因为“云盘真的一直在变”
- 而是因为：
  - 学习系统分散
  - 热缓存没有稳定兜住最贵的冷路径
  - 某些端，尤其移动端 WebDAV，仍然在重复 fallback

## 7. 当前实际效果判断

- `Docker` 端
  - 旧文件：总体是在加速
  - 新 `V2` 文件：能播了，但首帧阶段“预热和学习的收益还没完全兑现”，所以仍有减速感

- 移动端 `HTTP`
  - 部分加速，但受缓存与 fallback 影响，收益不稳定

- 移动端 `WebDAV`
  - 目前还不能说是有效加速
  - 更像是“已经有学习和预热，但没有真正压住反复 fallback 和断流问题”

一句话总结：

预热和学习理论上都在加速；当前 `Docker` 端部分生效，移动端尤其 `WebDAV` 端仍然存在重复冷启动和重复试错，所以体感上会像“有学习，但还没真正学会”。
